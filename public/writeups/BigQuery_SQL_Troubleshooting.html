Task 1 — Environment Setup

As in the previous BigQuery lab, the first step was pinning the data-to-insights public project to the BigQuery resource tree. BigQuery does not display public datasets by default, so navigating to Explorer → Add data → Star a project by name was necessary before any querying could begin.

The lab also introduced the BigQuery Query Validator — a real-time syntax checker embedded in the query editor. A red exclamation mark flags errors as you type; a green checkmark confirms the query is syntactically valid and ready to run. Understanding the difference between a validator error (syntax) and a logic error (wrong results despite no errors) was the central skill being tested throughout.

Key distinction: The query validator catches syntax errors before execution. It cannot catch logic errors — queries that run successfully but return incorrect or misleading results. That requires analytical thinking, not just reading the error message.

Task 2 — How Many Unique Visitors Completed Checkout?

The scenario: a new analyst has provided a series of queries to count unique visitors who reached the checkout confirmation page. Each one has a problem. My job was to identify the error in each version and explain the correct fix.

Error trail: ten broken queries, systematically fixed

Query version history — errors identified & fixed
✕
Typo in dataset name syntax
data-to-inghts instead of data-to-insights — BigQuery cannot resolve the table reference and throws a "Not found" error.
✕
Legacy SQL table reference syntax syntax
[project:dataset.table] is BigQuery's legacy SQL format. Standard SQL requires backtick notation: `project.dataset.table`. Mixing them causes a parse error.
✕
Empty SELECT clause syntax
SELECT FROM ... with no columns specified. The validator flags this immediately — SQL requires at least one column or * after SELECT.
✕
Missing comma between columns — creates an alias instead logic
SELECT fullVisitorId hits_page_pageTitle — without a comma, SQL treats hits_page_pageTitle as an alias for fullVisitorId, returning only one column. This is a silent logic error — no validator warning.
✕
No deduplication — visitors counted multiple times logic
COUNT(fullVisitorId) counts every row, not every unique visitor. A visitor who hits the checkout page three times is counted three times. The fix is COUNT(DISTINCT fullVisitorId).
✕
Missing GROUP BY with aggregation function syntax
Using COUNT() alongside a non-aggregated column (hits_page_pageTitle) without GROUP BY causes BigQuery to throw an error — it cannot determine how to group the unaggregated column.

The corrected query

After working through each broken version, the correct query combines COUNT(DISTINCT ...) for true deduplication, GROUP BY for the non-aggregated column, and a WHERE filter to isolate only the checkout confirmation page:

BigQuery Standard SQL — Unique Checkout Visitors ✓ correct 
SELECT
  COUNT(DISTINCT fullVisitorId) AS visitor_count,
  hits_page_pageTitle
FROM `data-to-insights.ecommerce.rev_transactions`
WHERE hits_page_pageTitle = "Checkout Confirmation"
GROUP BY hits_page_pageTitle;

The most instructive error in this task was the missing comma. SELECT fullVisitorId hits_page_pageTitle executes without any error — yet returns completely wrong results. It silently aliases one column instead of selecting two. This is a perfect example of why query results must always be critically evaluated, not just checked for errors.

Task 3 — Which Cities Had the Most Transactions?

The second business question required ranking cities by transaction volume, then calculating an average products-per-order metric. This task introduced two new error categories: incorrect aggregation function choice, and filtering aggregated results with WHERE instead of HAVING.

Error 1 — Using a raw column instead of SUM() for transactions

The initial query selected totals_transactions directly without aggregating it. Since the query uses GROUP BY geoNetwork_city, every non-aggregated column must either be in the GROUP BY or wrapped in an aggregation function. The fix was replacing the bare column reference with SUM(totals_transactions) to get the true total per city.

❌ Broken
totals_transactions — selects a raw column alongside a GROUP BY, causing a BigQuery error or incorrect per-row values rather than city totals.
✓ Fixed
SUM(totals_transactions) AS total_products_ordered — correctly aggregates all transaction values within each city group into a single total.

Error 2 — Filtering aggregated results with WHERE instead of HAVING

The most common SQL misconception among new analysts: trying to filter on a calculated aggregate field using WHERE. In SQL's execution order, WHERE runs before aggregation — so it cannot see values like avg_products_ordered that only exist after GROUP BY has run. BigQuery throws an error when you try.

BigQuery Standard SQL — broken version ✕ error 
SELECT
  geoNetwork_city,
  SUM(totals_transactions) / COUNT(DISTINCT fullVisitorId) AS avg_products_ordered
FROM `data-to-insights.ecommerce.rev_transactions`
GROUP BY geoNetwork_city
WHERE avg_products_ordered > 20  -- ✕ WHERE cannot reference aggregated fields
ORDER BY avg_products_ordered DESC;
BigQuery Standard SQL — corrected version ✓ correct 
SELECT
  geoNetwork_city,
  SUM(totals_transactions) AS total_products_ordered,
  COUNT(DISTINCT fullVisitorId) AS distinct_visitors,
  SUM(totals_transactions) / COUNT(DISTINCT fullVisitorId) AS avg_products_ordered
FROM `data-to-insights.ecommerce.rev_transactions`
GROUP BY geoNetwork_city
HAVING avg_products_ordered > 20  -- ✓ HAVING filters AFTER aggregation
ORDER BY avg_products_ordered DESC;

The WHERE vs HAVING distinction is one of the most frequently misunderstood rules in SQL. It comes down to execution order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. HAVING is the only clause that can filter on aggregated values — it runs after GROUP BY has collapsed the rows.

Calculated field: average products per order

The task also required creating a derived metric — average products ordered per visitor per city — by dividing one aggregate by another: SUM(totals_transactions) / COUNT(DISTINCT fullVisitorId). This pattern of computing ratios from two different aggregate functions is a core analytical SQL skill used constantly in business intelligence work.

Task 4 — Products per Category: Fixing COUNT and NULL Handling

The final task asked how many distinct products exist in each product category — a straightforward question that exposed two additional error types: using a non-distinct COUNT, and mishandling NULL values.

Error 1 — GROUP BY without any aggregation

The first broken query used GROUP BY on two columns with no aggregate functions in the SELECT. While BigQuery can execute this (it effectively deduplicates rows), it returns the raw list of (product, category) pairs without any count — completely failing to answer the question. The fix required wrapping the product name in COUNT(DISTINCT ...).

Performance note: Large GROUP BY operations without aggregation functions on millions of rows are an anti-pattern. Filter first, aggregate second — this is especially important in BigQuery where query costs are based on bytes scanned.

Error 2 — COUNT() vs COUNT(DISTINCT): the most critical aggregation mistake

The second broken version added COUNT(hits_product_v2ProductName) — but without DISTINCT. This counts every row that has a product name, including duplicates of the same product appearing in multiple sessions. The result vastly inflates the apparent product count per category.

❌ Inflated count
COUNT(hits_product_v2ProductName) — counts every row with a product name, including the same product viewed multiple times across many sessions.
✓ True distinct count
COUNT(DISTINCT hits_product_v2ProductName) — counts each unique product name exactly once per category, regardless of how many session rows it appears in.

Error 3 — NULL values polluting results

The dataset contained rows where the product name field was NULL — sessions where no product was recorded. Without filtering these out, they contribute a NULL "product" to the count and the results. Adding WHERE hits_product_v2ProductName IS NOT NULL before the GROUP BY ensures only real product records are counted.

BigQuery Standard SQL — Final correct query ✓ correct 
SELECT
  COUNT(DISTINCT hits_product_v2ProductName) AS number_of_products,
  hits_product_v2ProductCategory
FROM `data-to-insights.ecommerce.rev_transactions`
WHERE hits_product_v2ProductName IS NOT NULL  -- exclude NULL product rows
GROUP BY hits_product_v2ProductCategory
ORDER BY number_of_products DESC
LIMIT 5;

The results also revealed two data quality anomalies worth flagging: (not set) appearing as a category (indicating products with no assigned category in the CMS), and ${productitem.product.origCatName} appearing as a category — front-end template code that fired before the page fully loaded, causing the raw variable name to be captured by the analytics script instead of the actual category value. These are the kinds of upstream data quality issues that SQL alone cannot fix.

Key Takeaways

  • 1 A query that runs is not necessarily a query that is correct. Half the errors in this lab produced no validator warnings — they executed cleanly and returned results that looked plausible but were factually wrong. Validating results, not just execution, is essential analytical practice.
  • 2 COUNT vs COUNT DISTINCT is one of the most consequential choices in SQL. Using plain COUNT when you mean COUNT DISTINCT can inflate a metric by orders of magnitude on a dataset with millions of rows and repeat visitors. This distinction matters enormously for any visitor, customer, or product analysis.
  • 3 SQL execution order is not the same as write order. Understanding that WHERE → GROUP BY → HAVING run in sequence — and that HAVING is the only post-aggregation filter — is the most important conceptual model in SQL. Writing it correctly follows automatically once you understand the order.
  • 4 Missing commas are silent killers. A missing comma between column names creates an alias instead of a second column — no error, no warning, just subtly wrong output. This is why code review and result inspection matter even when the validator is green.
  • 5 NULL handling must be deliberate. NULL values propagate through aggregations in ways that corrupt results. Explicitly filtering with IS NOT NULL (or choosing IS NULL to investigate missing data) should be a standard step in any query that touches fields which may be sparsely populated.
  • 6 Data quality anomalies are visible in SQL results. The ${productitem.product.origCatName} value appearing in query results — raw front-end template code captured by analytics — is the kind of upstream tracking issue that SQL analysis surfaces first. Knowing how to spot and communicate these anomalies is a real analyst skill.
  • 7 The BigQuery Query Validator is a first-pass tool, not a complete safety net. It catches syntax errors instantly and is an invaluable accelerator. But logic errors, performance anti-patterns, and data quality problems require domain knowledge and careful result inspection — tools no validator can replace.
