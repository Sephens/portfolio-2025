Troubleshooting and Solving Data Join Pitfalls

A deep dive into the most dangerous mistakes when joining database tables — from many-to-many key relationships and unintentional row multiplication to choosing the right join type and surfacing hidden data gaps.

Overview

Of all the ways to corrupt data in SQL, join errors are among the most insidious. Unlike syntax mistakes that are caught immediately, join pitfalls often produce results that look plausible — the query runs, returns a number, and nobody questions it. Until someone notices the inventory figures are triple what they should be.

This intermediate lab used the Google Merchandise Store's ecommerce dataset to expose four of the most common join pitfalls in production SQL work: joining on non-unique keys, losing records to an incorrect join type, misreading NULL values, and triggering an unintentional cross join through a many-to-many relationship. For each pitfall, I diagnosed the root cause, quantified the damage, and implemented the correct solution.

Join Types: A Reference

Before diagnosing pitfalls, it's important to understand what each join type actually does to your data. The choice of join type is not cosmetic — it determines which rows survive the operation and which are discarded or multiplied.

INNER JOIN — Default
Returns only rows where the key exists in both tables. Rows with no match on either side are silently dropped. The most common source of missing data in analytics.
LEFT JOIN
Returns all rows from the left table, with matched data from the right table where available. Non-matching right-side values appear as NULL. Preserves your primary dataset.
RIGHT JOIN
The mirror of a LEFT JOIN — all rows from the right table are preserved. Rarely used in practice; most teams achieve the same result by swapping table order in a LEFT JOIN.
FULL JOIN
Returns all rows from both tables. Non-matching rows on either side appear with NULLs. Ideal for finding gaps and mismatches across two datasets simultaneously.
CROSS JOIN — Use with extreme caution
Produces every possible combination of rows from both tables — no join key required. With tables of M and N rows, the result is M × N rows. One extra row in either table multiplies the entire output. Legitimate for distributing a single lookup value (e.g., a discount rate) across all rows; dangerous when the source table unexpectedly grows.

Task 1 — Understanding Your Key Fields Before Joining

The foundation of any safe join is knowing the cardinality of your key fields — specifically, whether the key you plan to join on is unique in each table. Assuming uniqueness without verifying it is one of the most expensive mistakes in data work.

Discovering the SKU–Product Name relationship

The ecommerce dataset has two candidate fields for joining: productSKU and v2ProductName. I first queried both together to get a baseline count — returning 2,273 rows. I then queried for distinct SKUs alone and got 1,909. The difference revealed that multiple product names can share a single SKU.

BigQuery Standard SQL — SKU cardinality check
-- How many product names share the same SKU?
SELECT
  v2ProductName,
  COUNT(DISTINCT productSKU) AS SKU_count,
  STRING_AGG(DISTINCT productSKU LIMIT 5) AS SKU
FROM `data-to-insights.ecommerce.all_sessions_raw`
WHERE productSKU IS NOT NULL
GROUP BY v2ProductName
HAVING SKU_count > 1
ORDER BY SKU_count DESC;

The results confirmed a one-to-many relationship: one product name (like "Android Women's Short Sleeve Badge Tee") had up to 12 SKUs — one per size and colour variant. That's expected and by design.

The more dangerous discovery came next: querying in the reverse direction revealed that some single SKUs had multiple different product names. For example, GGOEGPJC019099 appeared under three names: "7" Dog Frisbee", "7" Dog Frisbee", and "Google 7-inch Dog Flying Disc Blue" — a special character variant, a punctuation variant, and an entirely different name for what was clearly the same physical product.

This is a many-to-many (N:N) key relationship — one SKU mapped to multiple names, and multiple SKUs mapped to multiple names. Joining two tables on an N:N key without deduplicating first is the root cause of most unintentional row multiplication in SQL analytics.

Pitfall 1 — Triple-Counting Inventory via a Non-Unique Key

With the N:N relationship identified, I next demonstrated what happens in practice when you join on that non-unique key. The product inventory table (ecommerce.products) held one clean record for GGOEGPJC019099 with a stock level of 154 units. Joining the website sessions table against this on the SKU field should return 154 — but it didn't.

The join that corrupts the count

Because the website sessions table had three rows for the same SKU (one per product name variant), the INNER JOIN matched the single inventory record against all three website rows — replicating the stock level three times.

BigQuery Standard SQL — the broken aggregation✕ wrong result
WITH inventory_per_sku AS (
  SELECT DISTINCT
    website.v2ProductName,
    website.productSKU,
    inventory.stockLevel
  FROM `data-to-insights.ecommerce.all_sessions_raw` AS website
  JOIN `data-to-insights.ecommerce.products` AS inventory
    ON website.productSKU = inventory.SKU
  WHERE productSKU = 'GGOEGPJC019099'
)
SELECT
  productSKU,
  SUM(stockLevel) AS total_inventory  -- sums 154 × 3 = 462
FROM inventory_per_sku
GROUP BY productSKU;
1 inventory record (stockLevel = 154)
× 3 website rows (same SKU, 3 different names)
= SUM reports 462 ✕
The inventory figure is inflated by a factor of 3 — a silent, dangerous error with no warning or error message

The fix: deduplicate the key before joining

The solution is to reduce the website dataset to one row per SKU before joining against the inventory table. ARRAY_AGG(DISTINCT v2ProductName LIMIT 1) collapses all name variants into a single row per SKU, eliminating the multiplication.

BigQuery Standard SQL — ARRAY_AGG deduplication✓ correct
-- Collapse all name variants into one row per SKU first
SELECT
  productSKU,
  ARRAY_AGG(DISTINCT v2ProductName LIMIT 1) AS product_name
FROM `data-to-insights.ecommerce.all_sessions_raw`
WHERE productSKU = 'GGOEGAAX0098'
GROUP BY productSKU;
-- Now each SKU appears exactly once — safe to join on
Rule: Before joining two tables, always verify the cardinality of the key in both tables. A 1:1 join is safe. A 1:N join requires awareness. An N:N join without deduplication will silently multiply your data.

Pitfall 2 — Silently Losing 819 Records with an INNER JOIN

The second pitfall was subtler — and arguably more common. When I joined the website sessions table against the product inventory table on SKU using the default INNER JOIN, I expected to get back all 1,909 distinct website SKUs. Instead, only 1,090 returned. The other 819 had disappeared silently.

No error. No warning. Just a result that was 43% smaller than the correct answer.

Why INNER JOIN drops records

An INNER JOIN only returns rows where the key exists in both tables. The 819 missing SKUs existed on the website — real products that real customers had viewed or purchased — but had no corresponding entry in the product inventory table. The INNER JOIN silently excluded them because its condition could not be satisfied.

BigQuery Standard SQL — diagnosing missing records with LEFT JOIN✓ correct join type
-- LEFT JOIN preserves ALL 1,909 website SKUs
SELECT DISTINCT
  website.productSKU   AS website_SKU,
  inventory.SKU        AS inventory_SKU  -- will be NULL for missing products
FROM `data-to-insights.ecommerce.all_sessions_raw` AS website
LEFT JOIN `data-to-insights.ecommerce.products` AS inventory
  ON website.productSKU = inventory.SKU;
-- Now filter to expose only the missing ones
-- WHERE inventory.SKU IS NULL → returns 819 rows

What the missing products actually were

Filtering for WHERE inventory.SKU IS NULL after the LEFT JOIN exposed all 819 products that existed on the website but not in the inventory system. This is a legitimate business scenario — not necessarily a data error. Possible explanations include digital products with no physical stock, products discontinued from inventory but still visible in historical order data, or simple gaps in inventory record-keeping. Crucially, an INNER JOIN would have hidden this gap entirely.

The default INNER JOIN is often the wrong choice for analytical queries. When your goal is to understand your full dataset — including gaps, mismatches, and missing records — always start with a LEFT JOIN from your primary table and use NULL filtering to expose what's missing. Never let the join type silently decide what you get to see.

Reversing the question: products in inventory but not on the website

I then used a RIGHT JOIN to ask the opposite question — which inventory products had never appeared on the website? Two records surfaced: a USB soundbar marked "in store only" (10 units ordered, 15 in stock), and a new PC gaming speaker with zero orders and no sentiment score. The gaming speaker's absence from web analytics made perfect sense — a brand-new product with no orders or page views yet would generate no analytics events. This is a real pattern in ecommerce data: inventory often gets ahead of web analytics.

The complete picture: FULL JOIN

A FULL JOIN combined both findings into a single query — all 819 website SKUs missing from inventory, plus the 2 inventory products missing from the website, totalling 821 unmatched records across both directions.

BigQuery Standard SQL — FULL JOIN for complete gap analysis
SELECT DISTINCT
  website.productSKU  AS website_SKU,
  inventory.SKU       AS inventory_SKU
FROM `data-to-insights.ecommerce.all_sessions_raw` AS website
FULL JOIN `data-to-insights.ecommerce.products` AS inventory
  ON website.productSKU = inventory.SKU
WHERE website.productSKU IS NULL
   OR inventory.SKU IS NULL;
-- Returns 821: 819 missing from inventory + 2 missing from website
819 website SKUs with no inventory record (LEFT JOIN gaps)
+ 2 inventory SKUs with no website record (RIGHT JOIN gaps)
= 821 total mismatches revealed by FULL JOIN ✓
LEFT JOIN + RIGHT JOIN = FULL JOIN — the complete picture of cross-table gaps

Pitfall 3 — The Unintentional Cross Join

The final and most explosive pitfall involved a CROSS JOIN — the join type that requires no matching key at all, combining every row in one table with every row in another. Used correctly, a CROSS JOIN is a clean way to apply a single lookup value (like a discount percentage) uniformly across an entire dataset. Used carelessly, it multiplies your output by whatever the size of the lookup table happens to be — including rows you didn't expect to be there.

The setup: a discount table

I created a simple one-row table containing a 5% site-wide discount rate and cross-joined it against all products in the Clearance category. With one discount row and 82 clearance products, the result was exactly 82 rows — correct and expected.

BigQuery Standard SQL — creating the discount table
CREATE OR REPLACE TABLE ecommerce.site_wide_promotion AS
SELECT .05 AS discount;

-- CROSS JOIN: no ON condition needed
-- 82 clearance products × 1 discount row = 82 rows ✓
SELECT DISTINCT
  productSKU,
  v2ProductCategory,
  discount
FROM `data-to-insights.ecommerce.all_sessions_raw` AS website
CROSS JOIN ecommerce.site_wide_promotion
WHERE v2ProductCategory LIKE '%Clearance%';

The moment it breaks

I then inserted two additional discount rows into the promotion table (0.04 and 0.03), bringing the total to three rows. Rerunning the exact same query — without changing a single character — now returned 246 rows instead of 82.

82 clearance products × 1 discount row = 82 rows ✓
82 clearance products × 3 discount rows = 246 rows ✕
The query didn't change. The data did. The CROSS JOIN silently tripled the output.
The danger of CROSS JOIN: The query itself gives no indication anything is wrong. The result set looks like product data. The numbers look like discount rates. Only someone who already knew the expected row count would catch the error — and in a production pipeline consuming this output downstream, the triple-counting could propagate silently through every dependent report.

The broader lesson

This is not a problem limited to explicit CROSS JOINs. Any join on an N:N key relationship produces the same effect — an implicit cross join where every matching row in one table pairs with every matching row in the other. The fix is the same in both cases: know the cardinality of your data before you join. Verify that lookup tables contain exactly one row for each key you intend to join on, and add assertions or row-count checks when those tables can change.

Key Takeaways

  • 1 Always verify key cardinality before writing a JOIN. The most destructive join errors come from assuming a key is unique when it isn't. A quick COUNT(*) vs COUNT(DISTINCT key) check takes seconds and can save hours of downstream debugging.
  • 2 INNER JOIN is almost never the right choice for analytical work. It silently discards any row without a match in both tables — dropping 819 records (43% of the dataset) without any error or warning. In analysis, gaps in data are often the most important finding. Use LEFT JOIN as your default and let NULL values surface the story.
  • 3 NULL values after a join are data, not noise. Every NULL in a LEFT JOIN result is a record that failed to match. Filtering for WHERE right_table.key IS NULL turns those NULLs into a precise, actionable list of missing records — a pattern directly applicable to data quality auditing and reconciliation work.
  • 4 Deduplicate before joining, not after. Using ARRAY_AGG(DISTINCT ... LIMIT 1) or a CTE to reduce a many-to-many key to a one-to-one key before joining prevents row multiplication entirely, rather than trying to correct for it in aggregation after the fact.
  • 5 CROSS JOINs are fragile by design — their output scales with their source tables. A CROSS JOIN against a lookup table that changes in size will silently change every downstream result. Any pipeline using a CROSS JOIN should either assert that the lookup table has exactly one row, or switch to a scalar subquery or a parameterised value that cannot unexpectedly multiply.
  • 6 LEFT JOIN + RIGHT JOIN = FULL JOIN. When you need a complete gap analysis across two datasets simultaneously — records missing in either direction — a FULL JOIN with NULL filtering on both sides is the cleanest single-query solution. It avoids running two separate queries and manually combining results.
  • 7 Data gaps have business explanations worth investigating. The 819 missing inventory SKUs and the 2 missing website SKUs were not simply data errors — they reflected real scenarios (digital products, new catalogue items, in-store-only stock). Good SQL analysis doesn't just surface anomalies; it provides the context to interpret them correctly.
