Permanent Tables vs Views — The Core Distinction

The choice between permanent tables and views is architectural — it determines how data is stored, how fresh it is, and what it costs to query. Understanding this difference is the prerequisite for all subsequent design decisions in BigQuery.

Permanent Table
Data physically stored in BigQuery. Query executes once, rows written to storage. Fast and cost-predictable to query. Does not auto-update — must be explicitly refreshed. Best for: large datasets queried frequently, expensive transformations you don't want to repeat, historical snapshots.
Logical View
Only the SQL query is stored — no data. Every query executes the underlying SELECT against live source tables. Always reflects current state. Query cost incurred on every access. Best for: always-current reporting, reusable complex logic, row-level security without duplicating data.

Use permanent tables when freshness can be traded for performance and cost predictability. Use views when real-time accuracy matters more than query-time cost, or when enforcing row-level security without maintaining separate copies of data for different user groups.

Task 2 — The Five Rules of CREATE TABLE in BigQuery

BigQuery's CREATE TABLE statement has five governing rules for how column definitions and query statements interact. Violating any of them produces an error — or worse, a query that succeeds but silently returns wrong data.

RuleDescriptionWhat breaks it
Rule 1Either a column list, a query statement, or both must be present.CREATE TABLE with neither a schema definition nor a SELECT clause.
Rule 2When both are present, BigQuery ignores query names and matches columns by position.Schema column A in position 1, query column B in position 1 — B's data silently populates A's column. No error.
Rule 3When only a query is present, BigQuery infers column names and types from the query output.Selecting a column without an alias may produce an unexpected inferred name.
Rule 4Column names must be defined either in the schema or the query statement.SELECT without aliases where BigQuery cannot infer a name.
Rule 5Duplicate column names are not allowed — in the schema definition or query output.SELECT fullVisitorId, * when * already includes fullVisitorId.

The complete error trail — six broken queries diagnosed

Query error catalogue — root cause & resolution for each
✕
Query 1 — Duplicate column name Rule 5
SELECT fullVisitorId, * — since * already includes fullVisitorId, the column appears twice. BigQuery rejects duplicate column names at table creation. Fix: use SELECT * alone, or list columns explicitly without repetition.
✕
Query 2 — Schema column count mismatch Rule 2
A 2-column schema was defined but SELECT * returns 32 columns. BigQuery cannot positionally match 2 schema columns to 32 query columns. Fix: match schema column count exactly to query output, or remove the schema and let BigQuery infer.
⚠
Query 3 — Valid syntax, silently wrong data Rule 2 — most dangerous
Schema defines channelGrouping in position 2, but the query selects city in position 2. BigQuery silently stores city data in the channelGrouping column. No error. No warning. The table is created and looks correct — but contains fundamentally wrong values.
✕
Query 4 — NOT NULL violation at runtime runtime error
totalTransactionRevenue INT64 NOT NULL passes validation but fails at execution — most visitor sessions have no purchase revenue, so NULL rows are the majority. Fix: remove NOT NULL from the revenue field, reflecting the business reality that it is optional.
✓
Query 5 — First correct execution working
NOT NULL on identifiers (always present), nullable revenue (optional by business logic). Executes successfully and creates the table with all 56,989 session records for 2017-08-01.
✓
Query 6 — Production revenue table: all rules correctly applied complete
Full schema with field descriptions, correct nullability, DISTINCT deduplication, CAST for type conversion, revenue divided by 1,000,000 as FLOAT64, WHERE filter for non-null revenue on a specific date. Returns exactly 97 distinct revenue-generating sessions.

The production revenue table — final correct query

BigQuery Standard SQL — production permanent table✓ complete
CREATE OR REPLACE TABLE ecommerce.revenue_transactions_20170801
-- Explicit schema: types, constraints, and descriptions defined upfront
(
  fullVisitorId           STRING  NOT NULL OPTIONS(description="Unique visitor ID"),
  visitId                 STRING  NOT NULL OPTIONS(description="Session ID — not globally unique"),
  channelGrouping         STRING  NOT NULL OPTIONS(description="Acquisition channel"),
  totalTransactionRevenue FLOAT64          OPTIONS(description="Transaction revenue in USD")
  -- nullable: not all visitors complete a purchase
)
OPTIONS(description="Revenue transactions for 08/01/2017")
AS
SELECT DISTINCT
  fullVisitorId,
  CAST(visitId AS STRING)       AS visitId,       -- INT64 → STRING for consistent key type
  channelGrouping,
  totalTransactionRevenue / 1000000 AS totalTransactionRevenue  -- micro-units → USD
FROM `data-to-insights.ecommerce.all_sessions_raw`
WHERE
  date = '20170801'
  AND totalTransactionRevenue IS NOT NULL;
-- Result: 97 revenue transactions
Stale data problem: Once created, a permanent table contains only the data ingested at creation time. Upstream changes are never automatically reflected. Pair permanent tables with BigQuery Scheduled Queries or Dataflow pipeline refresh jobs to keep them current.

Task 3 — Creating and Managing Views

Views solve the stale data problem. Because a view stores only the SQL query — not the data — every execution runs against live source tables, always returning the current state. The tradeoff: each access triggers the full underlying query and incurs query cost.

View creation modes

Naming convention: BigQuery has no visual indicator distinguishing tables from views in the Explorer panel. Always use a consistent prefix — vw_ prefix or _view suffix — to make the distinction immediately visible in any dataset listing, query, or documentation.
BigQuery Standard SQL — three CREATE VIEW modesreference
CREATE VIEW ecommerce.vw_name AS SELECT ...;
-- Creates only if view does NOT exist. Errors if it does.

CREATE OR REPLACE VIEW ecommerce.vw_name AS SELECT ...;
-- Creates or overwrites. Idempotent. Correct default for production.

CREATE VIEW IF NOT EXISTS ecommerce.vw_name AS SELECT ...;
-- Creates if absent; silently skips if exists. No error either way.

Views with metadata — labels, descriptions, and expiry

BigQuery Standard SQL — fraud detection view with full metadata✓ production pattern
CREATE OR REPLACE VIEW ecommerce.vw_large_transactions
OPTIONS(
  description          = "large transactions for review — orders over $1,000 USD",
  labels               = [('org_unit', 'loss_prevention')],
  expiration_timestamp = TIMESTAMP_ADD(CURRENT_TIMESTAMP(), INTERVAL 90 DAY)
  -- auto-expires: prevents sensitive views persisting indefinitely
)
AS
SELECT DISTINCT
  date, fullVisitorId, visitId, channelGrouping,
  totalTransactionRevenue / 1000000  AS revenue,
  currencyCode,
  STRING_AGG(DISTINCT v2ProductName ORDER BY v2ProductName LIMIT 10)
    AS products_ordered  -- collapses multiple product rows into one sorted string
FROM `data-to-insights.ecommerce.all_sessions_raw`
WHERE
  (totalTransactionRevenue / 1000000) >= 1000  -- must repeat expression — alias not yet defined
  AND currencyCode = 'USD'
  AND date >= '20170101'
GROUP BY 1,2,3,4,5,6   -- required when using STRING_AGG
ORDER BY date DESC
LIMIT 10;
Why WHERE repeats the division: SQL's execution order — FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY — means WHERE runs before SELECT. The alias revenue doesn't exist yet when WHERE evaluates. You must always repeat the full expression.

Task 4 — Row-Level Access Control with SESSION_USER()

The most technically sophisticated task: implementing domain-based row-level access control inside a view definition. SESSION_USER() returns the email address of the currently authenticated user at query time. Combined with REGEXP_EXTRACT to parse their domain and an IN filter to allowlist approved domains, it creates a view that returns different rows to different users — no separate table, no per-user IAM role required.

BigQuery Standard SQL — complete access-controlled view with expiry✓ enterprise pattern
CREATE OR REPLACE VIEW ecommerce.vw_large_transactions
OPTIONS(
  description          = "large transactions for review",
  labels               = [('org_unit', 'loss_prevention')],
  expiration_timestamp = TIMESTAMP_ADD(CURRENT_TIMESTAMP(), INTERVAL 90 DAY)
)
AS
SELECT DISTINCT
  SESSION_USER()                              AS viewer_ldap,
  REGEXP_EXTRACT(SESSION_USER(), r'@(.+)')  AS domain,
  date, fullVisitorId, visitId, channelGrouping,
  totalTransactionRevenue / 1000000           AS totalTransactionRevenue,
  currencyCode,
  STRING_AGG(DISTINCT v2ProductName ORDER BY v2ProductName LIMIT 10) AS products_ordered
FROM `data-to-insights.ecommerce.all_sessions_raw`
WHERE
  (totalTransactionRevenue / 1000000) > 1000
  AND currencyCode = 'USD'
  -- DOMAIN ALLOWLIST: users outside approved domains see zero rows
  AND REGEXP_EXTRACT(SESSION_USER(), r'@(.+)') IN ('qwiklabs.net')
  -- IN ('') → zero rows for ALL users — emergency kill switch
GROUP BY 1,2,3,4,5,6,7,8
ORDER BY date DESC
LIMIT 10;

BigQuery's full access control hierarchy

Project Level
IAM RolesBroadest scope. roles/bigquery.dataViewer grants access to all datasets in a project. Best for trusted team-wide platform access. Managed via Google Cloud IAM.
Dataset Level
Dataset IAMControls access to all tables and views within a specific dataset. Ideal for separating restricted raw data from curated reporting datasets with broader access.
View Level
Authorized ViewsExpose a curated subset of a restricted dataset without granting direct table access. The view holds its own IAM permissions separate from the underlying tables.
Row Level
SESSION_USER() + Row Access PoliciesFinest granularity. SESSION_USER() inside views filters rows based on caller identity at query time. BigQuery's native Row Access Policies (GA since 2021) achieve the same directly on tables using CREATE ROW ACCESS POLICY DDL — the recommended modern production approach.

The SESSION_USER() pattern in this lab is the conceptual foundation for BigQuery's production Row Access Policies. Modern implementations attach policies directly to the base table — making security enforcement invisible to downstream views and impossible to circumvent by querying the source table directly.

Key Takeaways

  • 1BigQuery matches schema columns to query output by position, not by name. When both a schema and a query statement are present, query column names are completely ignored. A positional mismatch silently populates wrong columns — one of the most dangerous failure modes because it produces no error.
  • 2NOT NULL is a data quality enforcement gate, not just documentation. It creates a hard enforcement point that rejects upstream data missing a required field at write time rather than silently storing NULL. Use it on fields where a NULL would make the row meaningless.
  • 3Permanent tables and views solve different problems. Tables: fast, cost-predictable queries on static data. Views: always-current data at the cost of re-executing the full query on every access. Choose based on whether freshness or performance matters more for each use case.
  • 4CREATE OR REPLACE VIEW is the correct default for production pipelines. It is idempotent — safe to run repeatedly in automated deployment scripts and scheduled jobs without risk of failure if the view already exists.
  • 5WHERE clauses cannot reference aliased columns defined in SELECT. SQL's execution order means WHERE runs before SELECT. Always repeat the full expression — the alias defined in SELECT does not exist yet when WHERE is evaluated.
  • 6SESSION_USER() enables row-level security baked directly into view definitions. By filtering rows based on the authenticated caller's domain at query time, one view returns different results to different users without separate tables, datasets, or per-user IAM configurations.
  • 7View expiration timestamps prevent sensitive data persisting indefinitely. Adding expiration_timestamp to view OPTIONS ensures it self-destructs after the retention window — essential security hygiene for views exposing sensitive financial or personal data.
  • 8Labels are infrastructure metadata, not UI decoration. Labels on views and tables enable programmatic filtering in Cloud Console, cost attribution in billing exports, and automated governance audits across an entire GCP project.
