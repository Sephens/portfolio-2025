Part 1 — The Five Principles of Dataset Integrity

High-quality datasets conform to five strict rules. Each represents a different dimension of data trustworthiness. Understanding all five — and recognising which one is violated in a given dataset — is the first skill of any serious data practitioner.

1Validity
Data conforms to defined business rules and structural constraints — correct data types, allowed value ranges, required fields, and format patterns.
2Accuracy
Data matches a real-world source of truth. A value can be structurally valid (correct type, correct range) yet still be factually wrong.
3Completeness
No required data is missing. You are seeing the whole picture — no rows filtered, no fields truncated, no subsets masquerading as totals.
4Consistency
The same fact is represented the same way across all systems. No two data sources contradict each other for the same entity.
5Uniformity
All data uses the same units, formats, and standards throughout. Mixing imperial and metric, or different date formats, breaks comparability.

Principle 1: Validity — Does the data conform to the rules?

Validity is about structural correctness. A value is valid if it satisfies every constraint defined for its field. These constraints fall into eight categories:

  • Data-type constraints An age field must contain integers, not strings like "old".
  • Range constraints A single die roll must be 1–6. A two-dice roll must be 2–12. A value of 1 is invalid for two dice.
  • Mandatory constraints Customer ID and email fields cannot be NULL.
  • Set-membership constraints A country_code field must contain only valid ISO 3166 codes.
  • Foreign-key constraints An order's customer_id must reference a real customer record.
  • Regex patterns An email field must match [a-z]+@[a-z]+\.[a-z]+.
  • Cross-field validation A ship_date cannot be earlier than an order_date for the same record.
  • Unique constraints No two customers should share the same phone number, licence plate, or mailing address.

Validity failures are most common in legacy systems and spreadsheet-based data capture, where there was no software enforcement of constraints. Modern databases prevent most validity issues at ingestion — but you still need to validate incoming data from external sources.

Principle 2: Accuracy — Does the data reflect the real world?

A value can be completely valid — correct type, in range, not null — yet still be factually wrong. A US state field that contains "Hot Dog" is a valid string, but it doesn't match any entry in the known list of 50 US states. Accuracy requires an external source of truth to validate against.

This makes accuracy the hardest principle to enforce through SQL alone. It typically requires a JOIN against a reference dataset — a curated list of valid states, countries, product codes, or financial instruments. Without that reference, you can check that a value looks right; you cannot confirm that it is right.

Important nuance: Sources of truth can themselves vary. Some datasets include US territories like Puerto Rico and Guam alongside states; others list only the 50 states. Accuracy is always relative to a defined standard — which is why documenting your source of truth is as important as enforcing it.

Principle 3: Completeness — Do you have the whole picture?

Completeness is the principle most commonly violated without anyone noticing. We instinctively trust data that arrives in row-and-column format — we assume that if a table exists, it contains everything we need. That assumption is dangerous.

Consider being shown two close-up photographs of a city street — a decorative lamp and a clock tower — without context. From those fragments alone, identifying the city is nearly impossible. But zoom out to the full panorama and it is immediately recognisable as London. Incomplete data produces the same distortion: analysis that is technically accurate over the available subset but fundamentally misleading about the whole.

Consequences of incomplete data include: biased results that over-represent the sampled segment; patterns that appear meaningful but are artefacts of missing data; and machine learning models that overfit to a fragment rather than generalising across the population.

Principle 4: Consistency — Do all systems agree?

Inconsistency occurs when two data items within or across systems contradict each other. Consider: System A records the house at 123 ABC Street as owned by Owner ID 12. System B records Owner ID 12 at a completely different address, and 123 ABC Street as owned by Owner ID 15. Both cannot be right. This is a referential integrity failure — and it means the joined view of the data cannot be trusted.

Resolving inconsistency is not always possible through SQL. It may require determining which system is the authoritative source, checking which record was most recently updated, or external verification. SQL can surface inconsistencies; it cannot always arbitrate between them.

Principle 5: Uniformity — Is everything measured the same way?

Uniformity failures occur when the same dimension is measured in different units across different parts of a system. The consequences can be catastrophic. On November 10, 1999, NASA's Mars Climate Orbiter — a $125 million spacecraft — was lost because one engineering team used imperial units (pounds-force seconds) while another used metric (newton-seconds). The mismatch sent the probe on the wrong trajectory and it burned up in the Martian atmosphere.

In data, the same principle applies: mixing metric and imperial measurements, mixing date formats (DD/MM/YYYY vs MM/DD/YYYY), or mixing time zones across event logs will silently corrupt every calculation that combines them.

Uniformity failures are especially common when data is ingested from multiple geographic regions, legacy systems with different formatting standards, or third-party providers with different conventions. Document the standard before you start; enforce it with SQL functions at ingestion.

Dataset Shape and Skew

Beyond the five integrity principles, every dataset has a shape (the ratio of rows to columns) and a skew (the distribution of values within it). Both affect the reliability of analysis.

A wide-but-short dataset has many columns but few rows — you have rich attributes but not enough observations to draw statistically sound conclusions. A tall-but-narrow dataset has many rows but few columns — plenty of observations, but limited context per observation. The ideal is a dataset with enough columns to capture the phenomenon of interest and enough rows to make defensible inferences.

Skew is the concentration of values. If an international sales dataset shows 95% of orders from one country, you face a choice: is that an accurate reflection of the business, or is it a data collection gap? Skew is not inherently bad — a business genuinely concentrated in one market will show skewed data. The question is always whether the skew reflects objective reality or is an artefact of missing or biased collection.

How to detect skew in BigQuery: Run a GROUP BY on the suspected dimension and ORDER BY COUNT(*) DESC. If one value dominates dramatically, investigate the collection process before proceeding with analysis.

Part 2 — Cleaning and Transforming Data with SQL

SQL is the primary tool for data cleaning in BigQuery. The following techniques map directly onto the five integrity principles, giving you a concrete, executable response to each type of data quality problem.

Handling NULL Values Beginner

NULL in SQL represents the complete absence of a value — it is not zero, not an empty string, not the word "null". It is the absence of data itself. NULL values propagate: any arithmetic operation involving a NULL returns NULL. Any comparison using = NULL returns NULL, not TRUE or FALSE. This is why you must use IS NULL and IS NOT NULL, never = NULL.

BigQuery Standard SQL — NULL detectionbeginner
-- Find all rows with a missing email address
SELECT *
FROM `project.dataset.customers`
WHERE email IS NULL;

-- Count how many rows have missing values per column
SELECT
  COUNTIF(email IS NULL)     AS missing_email,
  COUNTIF(phone IS NULL)     AS missing_phone,
  COUNTIF(country IS NULL)   AS missing_country,
  COUNT(*)                    AS total_rows
FROM `project.dataset.customers`;

The NULL-handling function family

COALESCE(a, b, c, …)
Returns the first non-NULL argument. Accepts any number of arguments. The Swiss Army knife of NULL handling.
IFNULL(value, replacement)
Returns value if it is not NULL, otherwise returns replacement. Simpler version of COALESCE for exactly two options.
NULLIF(value, match)
Returns NULL if value equals match; otherwise returns value. Use to treat empty strings as NULLs.
BigQuery Standard SQL — NULL replacement patternsbeginner
-- IFNULL: substitute a default when a value is missing
SELECT
  customer_id,
  IFNULL(phone, 'Not provided') AS phone_clean
FROM `project.dataset.customers`;

-- COALESCE: try multiple fallback columns in order
SELECT
  order_id,
  COALESCE(delivery_email, billing_email, account_email, 'unknown@placeholder.com')
    AS best_contact_email
FROM `project.dataset.orders`;

-- NULLIF: treat blank strings as true NULLs
-- An empty string '' passes IS NOT NULL checks — this fixes that
SELECT
  customer_id,
  NULLIF(state, '') AS state_clean  -- blank → NULL
FROM `project.dataset.customers`;
The blank string trap: A BigQuery column showing what appears to be an empty value may actually contain an empty string '' — not a NULL. These are completely different in SQL. An empty string is a valid data value that will pass IS NOT NULL checks and appear in counts. Use NULLIF(column, '') to convert blank strings to true NULLs, or add AND column != '' to your WHERE clause.

Removing Duplicate Records Intermediate

Duplicate rows silently inflate every metric — totals, averages, counts, conversion rates. There are two main approaches to deduplication in BigQuery: identifying duplicates with GROUP BY + HAVING (for detection), and using ROW_NUMBER() window function (for surgical removal, keeping the best version of each duplicate).

BigQuery Standard SQL — finding and removing duplicatesintermediate
-- STEP 1: Detect duplicates
SELECT
  customer_id,
  email,
  COUNT(*) AS duplicate_count
FROM `project.dataset.customers`
GROUP BY customer_id, email
HAVING duplicate_count > 1;

-- STEP 2: Keep only the most recently updated version
-- ROW_NUMBER() assigns 1 to the "best" row in each duplicate group
WITH ranked_customers AS (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY customer_id        -- group duplicates by key
      ORDER BY     updated_at DESC   -- most recent first
    ) AS row_num
  FROM `project.dataset.customers`
)
SELECT * EXCEPT(row_num)
FROM   ranked_customers
WHERE  row_num = 1;   -- keep only the best row per customer
When to use which approach: Use GROUP BY + HAVING to audit and report on duplicates. Use ROW_NUMBER() when you need to preserve one record per group and have a clear ranking criterion (most recent, highest value, etc.). Never blindly delete duplicates without understanding which version is canonical.

Conditional Validation with CASE WHEN Beginner

CASE WHEN is SQL's conditional branching construct — equivalent to if/else in programming. In data cleaning, it is used to classify, flag, recode, and correct values based on rules. It is one of the most versatile tools in the SQL cleaning toolkit.

BigQuery Standard SQL — CASE WHEN for validation and recodingbeginner
-- Flag records that fail a range validation check
SELECT
  order_id,
  quantity,
  unit_price,
  CASE
    WHEN quantity < 0        THEN 'Invalid: negative quantity'
    WHEN unit_price <= 0    THEN 'Invalid: zero or negative price'
    WHEN quantity > 10000   THEN 'Suspect: unusually large order'
    ELSE                          'Valid'
  END AS validation_status
FROM `project.dataset.orders`;

-- Recode inconsistent category labels into a standard set
SELECT
  product_id,
  CASE
    WHEN LOWER(category) IN ('electronics', 'electronic', 'electr.') THEN 'Electronics'
    WHEN LOWER(category) IN ('clothing', 'apparel', 'clothes')   THEN 'Apparel'
    WHEN category IS NULL                                         THEN 'Uncategorised'
    ELSE category
  END AS category_clean
FROM `project.dataset.products`;

String Cleaning Functions Intermediate

Inconsistent string formatting — mixed case, leading/trailing spaces, special characters, encoding artefacts — is one of the most common sources of data quality failures. A customer named "John Smith" and "john smith" and " John Smith " are the same person but will appear as three distinct records in a GROUP BY. The following functions solve this:

BigQuery Standard SQL — string standardisationintermediate
-- TRIM removes leading and trailing whitespace
-- LOWER standardises case for comparison
-- REPLACE swaps out known bad characters
SELECT
  customer_id,
  TRIM(LOWER(first_name))              AS first_name_clean,
  TRIM(LOWER(last_name))               AS last_name_clean,
  REPLACE(phone, '-', '')             AS phone_no_dashes,
  REGEXP_REPLACE(phone, r'[^0-9]', '') AS phone_digits_only
FROM `project.dataset.customers`;
BigQuery Standard SQL — REGEXP_REPLACE for pattern-based cleaningintermediate
-- Remove all non-alphanumeric characters from a product SKU field
-- Pattern r'[^A-Z0-9]' matches anything that is NOT A-Z or 0-9
SELECT
  product_id,
  productSKU,
  REGEXP_REPLACE(productSKU, r'[^A-Z0-9]', '') AS sku_clean
FROM `project.dataset.products`;

-- Standardise US phone numbers to (XXX) XXX-XXXX format
-- First strip all non-digits, then reformat
SELECT
  customer_id,
  CASE
    WHEN REGEXP_CONTAINS(
      REGEXP_REPLACE(phone, r'[^0-9]', ''),
      r'^\d{10}$'                    -- exactly 10 digits
    ) THEN
      CONCAT(
        '(', SUBSTR(REGEXP_REPLACE(phone, r'[^0-9]', ''), 1, 3), ') ',
             SUBSTR(REGEXP_REPLACE(phone, r'[^0-9]', ''), 4, 3), '-',
             SUBSTR(REGEXP_REPLACE(phone, r'[^0-9]', ''), 7, 4)
      )
    ELSE NULL                         -- invalid phone → NULL
  END AS phone_formatted
FROM `project.dataset.customers`;

Type Casting and Format Standardisation Intermediate

Data ingested from external sources often arrives with incorrect types — dates stored as strings, numbers stored as text, booleans stored as 0/1 integers. Operating on wrong types produces silent errors or incorrect results. CAST converts between types explicitly; SAFE_CAST returns NULL instead of throwing an error when conversion fails.

BigQuery Standard SQL — CAST and SAFE_CASTintermediate
-- CAST: convert string dates and string numbers to correct types
SELECT
  order_id,
  CAST(order_date_string AS DATE)              AS order_date,
  CAST(revenue_string    AS FLOAT64)           AS revenue,
  CAST(quantity_string   AS INT64)             AS quantity
FROM `project.dataset.raw_orders`;

-- SAFE_CAST: returns NULL instead of failing on bad values
-- Use this when you expect some values may not convert cleanly
SELECT
  record_id,
  SAFE_CAST(price_field AS FLOAT64) AS price_clean,
  -- If price_field = 'N/A' or 'pending', SAFE_CAST returns NULL
  -- CAST would throw an error
  COALESCE(SAFE_CAST(price_field AS FLOAT64), 0.0) AS price_with_default
FROM `project.dataset.raw_prices`;

Date and Time Standardisation Intermediate

Date and time inconsistencies are among the most common cleaning tasks. Different source systems use different formats (DD/MM/YYYY vs MM-DD-YY), different time zones (UTC vs local), or Unix epoch timestamps (seconds since 1 January 1970). BigQuery has a rich set of date/time functions to normalise all of these.

BigQuery Standard SQL — date/time cleaning and normalisationintermediate
-- Parse a string date in a non-standard format into a DATE type
SELECT
  PARSE_DATE('%d/%m/%Y', date_string)  AS parsed_date
FROM `project.dataset.raw_events`;

-- Convert UTC timestamps to Nairobi local time (EAT = UTC+3)
SELECT
  event_id,
  event_timestamp,
  DATETIME(event_timestamp, 'Africa/Nairobi') AS event_local_time
FROM `project.dataset.events`;

-- Convert Unix epoch (seconds since 1970-01-01) to readable TIMESTAMP
SELECT
  user_id,
  created_at_epoch,
  TIMESTAMP_SECONDS(created_at_epoch) AS created_at_readable
FROM `project.dataset.users`;

-- Extract components from a date for grouping and filtering
SELECT
  EXTRACT(YEAR  FROM order_date) AS order_year,
  EXTRACT(MONTH FROM order_date) AS order_month,
  EXTRACT(DAYOFWEEK FROM order_date) AS day_of_week,
  COUNT(*) AS orders
FROM `project.dataset.orders`
GROUP BY 1, 2, 3
ORDER BY 1, 2;

Validating Accuracy with Reference Joins Intermediate

Accuracy validation requires comparing your data against a trusted reference dataset. The cleanest SQL pattern for this is a LEFT JOIN against the reference table, then filtering for NULLs to expose values that don't match any valid entry — the same NULL-filtering pattern used in join pitfall analysis, applied here for data quality.

BigQuery Standard SQL — accuracy validation via LEFT JOINintermediate
-- Validate that all country_codes in orders match a known reference list
SELECT
  o.order_id,
  o.country_code,
  -- If ref.country_code is NULL, the value doesn't exist in the reference
  CASE WHEN ref.country_code IS NULL THEN 'INVALID' ELSE 'VALID' END AS validity_flag
FROM `project.dataset.orders` o
LEFT JOIN `project.reference.valid_country_codes` ref
  ON o.country_code = ref.country_code
WHERE ref.country_code IS NULL;  -- show only invalid entries

-- Alternatively: use a subquery with the IN operator
SELECT order_id, state
FROM `project.dataset.orders`
WHERE state NOT IN (
  SELECT state_code
  FROM `project.reference.us_states`
);

Enriching Data for Completeness Intermediate

Completeness is not just about fixing what's missing — it's also about augmenting your dataset with additional sources to get a fuller picture. This is the exact same JOIN and UNION pattern used in exploratory analysis, applied here with a data quality goal: adding context that makes your dataset more complete.

BigQuery Standard SQL — enrichment via JOIN and UNIONintermediate
-- Enrich orders with customer demographics from a separate system
SELECT
  o.order_id,
  o.revenue,
  o.order_date,
  c.customer_segment,    -- added from CRM
  c.country,             -- added from CRM
  c.age_band             -- added from CRM
FROM `project.dataset.orders`        o
LEFT JOIN `project.crm.customers`    c
  ON o.customer_id = c.customer_id;

-- Combine multi-year datasets via UNION ALL for temporal completeness
SELECT * FROM `project.weather.readings_2023`
UNION ALL
SELECT * FROM `project.weather.readings_2024`
UNION ALL
SELECT * FROM `project.weather.readings_2025`;

Cross-System Consistency Checks Advanced

Detecting consistency failures requires querying two systems simultaneously and looking for contradictions. The most powerful pattern is a FULL JOIN on the shared key, then using CASE expressions to flag where the two systems disagree.

BigQuery Standard SQL — cross-system consistency validationadvanced
-- Compare customer addresses between CRM and billing system
-- Flag any record where the two systems disagree
SELECT
  COALESCE(crm.customer_id, billing.customer_id) AS customer_id,
  crm.address                  AS crm_address,
  billing.address              AS billing_address,
  CASE
    WHEN crm.customer_id IS NULL     THEN 'Missing from CRM'
    WHEN billing.customer_id IS NULL THEN 'Missing from Billing'
    WHEN crm.address != billing.address THEN 'Address Mismatch'
    ELSE                               'Consistent'
  END AS consistency_status
FROM `project.crm.customers`         crm
FULL JOIN `project.billing.customers`  billing
  ON crm.customer_id = billing.customer_id
WHERE
  crm.address != billing.address      -- mismatches
  OR crm.customer_id IS NULL          -- missing from CRM
  OR billing.customer_id IS NULL;     -- missing from billing

A Production-Ready Cleaning Pipeline with CTEs Advanced

In production, cleaning steps are almost never applied in a single query. They are built as a staged pipeline — each CTE handles one concern, and the final SELECT produces the clean, enriched, validated output. This pattern is readable, testable, and repeatable when new dirty data arrives.

BigQuery Standard SQL — full multi-stage cleaning pipelineadvanced
-- Stage 1: Fix types, blank strings, and format issues
WITH stage_1_type_fix AS (
  SELECT
    customer_id,
    TRIM(LOWER(email))                            AS email,
    NULLIF(TRIM(phone), '')                       AS phone,
    SAFE_CAST(signup_date_str AS DATE)             AS signup_date,
    SAFE_CAST(lifetime_value_str AS FLOAT64)       AS lifetime_value,
    REGEXP_REPLACE(country_code, r'[^A-Z]', '')  AS country_code
  FROM `project.raw.customers`
),

-- Stage 2: Apply business-rule validations
stage_2_validated AS (
  SELECT
    *,
    CASE
      WHEN email IS NULL                           THEN 'FAIL: missing email'
      WHEN NOT REGEXP_CONTAINS(email, r'@.+\..+') THEN 'FAIL: invalid email format'
      WHEN signup_date > CURRENT_DATE()            THEN 'FAIL: future signup date'
      WHEN lifetime_value < 0                      THEN 'FAIL: negative value'
      ELSE                                               'PASS'
    END AS validation_status
  FROM stage_1_type_fix
),

-- Stage 3: Deduplicate — keep the most recently created record per email
stage_3_deduplicated AS (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY email
      ORDER BY     signup_date DESC
    ) AS rn
  FROM stage_2_validated
  WHERE validation_status = 'PASS'
),

-- Stage 4: Enrich with country reference data
stage_4_enriched AS (
  SELECT
    d.* EXCEPT(rn),
    ref.country_name,
    ref.region,
    ref.currency_code
  FROM      stage_3_deduplicated          d
  LEFT JOIN `project.reference.countries` ref
    ON d.country_code = ref.iso_code
  WHERE d.rn = 1
)

-- Final output: clean, validated, deduplicated, enriched
SELECT * FROM stage_4_enriched;

A multi-stage CTE pipeline like this is the gold standard for production data cleaning. Each stage has a single responsibility. Failures are easy to isolate — comment out downstream stages to inspect any intermediate result. And because it's a SELECT (not a DELETE), your raw data is never touched. The logic is repeatable every time new dirty data arrives.

Part 3 — Google Cloud ETL Tools Beyond SQL

While SQL in BigQuery handles the majority of data cleaning tasks, there are scenarios where SQL alone is insufficient — large-scale streaming pipelines, legacy system migrations, complex multi-source orchestrations, or teams without SQL expertise. Google Cloud provides five specialised tools for these scenarios.

Raw Sources → Extract → Transform / Clean → Load → BigQuery → Analysis / ML

Cloud Data Fusion & Wrangler

Cloud Data Fusion is a fully managed, code-free enterprise data integration platform. It provides a drag-and-drop graphical interface for building, deploying, and managing ETL/ELT pipelines — making it accessible to data analysts and business users who may not write code. Its core value is speed: pipelines that would take weeks to build in code can be assembled in hours.

Its embedded Wrangler tool is particularly powerful for data cleaning. It allows you to load a 10 MB sample of your data, apply transformations interactively using point-and-click operations, preview results instantly, and then scale the same transformation recipe to the full dataset. Wrangler is the right tool when you need to explore and experiment before committing to a transformation logic.

Best for: Teams without strong coding skills; multi-source integration (databases, Cloud Storage, Kafka); pipelines that need enterprise governance, lineage tracking, and metadata management; organisations needing a centralised, monitored data integration platform.

Dataflow (Apache Beam)

Dataflow is Google's fully managed, serverless data processing service for both stream processing (real-time) and batch processing (scheduled). It is built on the open-source Apache Beam framework, which means pipelines written for Dataflow can also run on other Beam-compatible runners.

The key concept in Dataflow is the pipeline — a directed graph of transformations where data flows through a series of stages: read from a source, apply transforms (filter, enrich, aggregate, join), write to a destination. The pipeline model abstracts away distributed computing complexities: sharding, worker coordination, failure recovery, and auto-scaling are all managed automatically.

Apache Beam (Python SDK) — Dataflow pipeline conceptadvanced
# Conceptual example of an Apache Beam pipeline
# This illustrates the logical flow — not BigQuery SQL
import apache_beam as beam

with beam.Pipeline() as pipeline:
    clean_records = (
        pipeline
        | 'Read'          >> beam.io.ReadFromBigQuery(query='SELECT * FROM raw.events')
        | 'Filter Nulls'  >> beam.Filter(lambda r: r['user_id'] is not None)
        | 'Normalise'    >> beam.Map(lambda r: {**r, 'email': r['email'].lower().strip()})
        | 'Write'        >> beam.io.WriteToBigQuery('project.dataset.clean_events')
    )
Best for: Real-time/streaming data (IoT sensors, clickstreams, fraud detection); large-scale batch jobs requiring complex transforms; teams with Python or Java engineering expertise; when you need the maximum flexibility of code-driven transformation logic.

Dataprep (by Trifacta)

Dataprep is a third-party intelligent data wrangling service operated by Trifacta, deeply integrated with Google Cloud. It is UI-based, auto-scales using Dataflow under the hood, and incorporates AI-driven suggestions for transformations. Unlike Wrangler (which is part of a broader pipeline tool), Dataprep is specifically focused on the cleaning and preparation step.

Dataprep can ingest data from BigQuery, Cloud Storage, and local files in CSV, JSON, or relational formats. Cleaned data can be sent directly to BigQuery, Cloud Storage, Looker Studio, or Vertex AI — making it a natural step in an end-to-end analytics or ML workflow.

Best for: Analysts who want a visual, AI-assisted interface for exploring and cleaning data; marketing analytics combining data from multiple cross-functional sources; data onboarding from external/third-party providers; preparation steps for ML model training in Vertex AI.

Dataproc

Dataproc is Google Cloud's managed service for running Apache Spark, Apache Hadoop, Apache Flink, and 30+ other open-source big data frameworks. It is the tool of choice when your organisation already has investment in Spark or Hadoop — either in terms of existing job code, engineering expertise, or data architecture dependencies.

Dataproc creates managed clusters that can be spun up in 90 seconds, scaled dynamically, and shut down when not in use — avoiding the cost of always-on infrastructure. It integrates natively with BigQuery, Cloud Storage, and Cloud Bigtable, allowing Spark jobs to read and write to these services directly.

Best for: Organisations with existing Spark/Hadoop expertise or job dependencies; large-scale ETL workloads at petabyte scale; data lake modernisation; ML feature engineering at scale; teams that want hands-on DevOps control over their processing cluster.

Key Takeaways

  • 1 Data quality is not a pre-processing step — it is the foundation of every analysis. The most sophisticated SQL query, the most beautiful visualisation, and the most accurate machine learning model are all worthless if built on data that violates the five integrity principles. Invest in cleaning before you invest in analysis.
  • 2 NULL and empty string are not the same thing — ever. An empty string passes IS NOT NULL, appears in counts, and shows up in GROUP BY results. Always use NULLIF(column, '') to convert blanks to true NULLs, or add AND column != '' to filter both simultaneously.
  • 3 COALESCE is the most versatile NULL-handling function. It accepts any number of arguments and returns the first non-NULL value — making it ideal for implementing fallback chains across multiple potential source columns. Master it early.
  • 4 Use SAFE_CAST instead of CAST when ingesting from external sources. External data will sometimes have values that can't be cast — text in a numeric field, invalid dates. SAFE_CAST returns NULL on failure; CAST throws a hard error that breaks the entire query.
  • 5 Multi-stage CTE pipelines are the gold standard for production cleaning. Each CTE has one responsibility: type fixing, validation, deduplication, enrichment. This makes pipelines debuggable (inspect any intermediate stage), readable (self-documenting), and repeatable (just re-run the query on new data).
  • 6 Skew is not always a problem — but it always requires investigation. A dataset where 90% of records come from one country may reflect the true business reality. Or it may reflect a broken data collection process. The answer determines whether you proceed or escalate. Never assume.
  • 7 The right Google Cloud ETL tool depends on your team's skills and your data's characteristics. SQL for in-BigQuery transformations; Data Fusion or Dataprep for visual/no-code pipelines; Dataflow for real-time streaming or maximum flexibility; Dataproc if your team already lives in Spark or Hadoop. There is no universally correct answer — only the correct answer for your context.
  • 8 Data cleaning logic must be repeatable, not one-off. Dirty data is not a one-time problem — it arrives continuously from imperfect source systems. Cleaning logic written as SQL CTEs or pipeline recipes can be scheduled, automated, and applied to every new batch of incoming data without manual intervention.
