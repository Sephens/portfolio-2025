Task 1 — Environment Setup & Dataset Access

BigQuery public datasets are not visible by default in the console. The first task involved navigating the BigQuery Explorer and pinning the data-to-insights project — a public repository hosted by Google containing curated datasets for learning and experimentation.

Once starred, I navigated to the ecommerce dataset and opened the all_sessions_raw table. I explored the three metadata views available in BigQuery's UI:

Schema Tab
Field names, data types (STRING, INTEGER, FLOAT), and field modes — the structural blueprint of the table.
Details Tab
Table metadata including row count (21M+), storage size (5.63 GB), creation date, and table ID.
Preview Tab
A live sample of actual row data — useful for building intuition about the content before writing queries.

Key learning: Understanding a dataset's schema and metadata before querying it is a fundamental best practice. It prevents type mismatches, informs your aggregation strategy, and helps identify fields that may contain nulls or require casting.

Task 2 — Data Quality: Identifying Duplicate Records

Before any analysis can be trusted, the underlying data must be validated. The all_sessions_raw table contained no single field that uniquely identified each row — meaning duplicates could only be detected by evaluating the combination of all fields together.

The Approach

I used a GROUP BY across every column in the table combined with COUNT(*). If every field in a row is truly unique, the count for that group will be exactly 1. Any group with a count greater than 1 represents a duplicate — rows that are identical across all fields. The HAVING clause then filters to show only those groups.

BigQuery Standard SQL — Duplicate Detection
SELECT COUNT(*) AS num_duplicate_rows, *
FROM `data-to-insights.ecommerce.all_sessions_raw`
GROUP BY
  fullVisitorId, channelGrouping, time, country, city,
  totalTransactionRevenue, transactions, timeOnSite,
  pageviews, sessionQualityDim, date, visitId, type,
  productRefundAmount, productQuantity, productPrice,
  productRevenue, productSKU, v2ProductName,
  v2ProductCategory, productVariant, currencyCode,
  itemQuantity, itemRevenue, transactionRevenue,
  transactionId, pageTitle, searchKeyword,
  pagePathLevel1, eCommerceAction_type,
  eCommerceAction_step, eCommerceAction_option
HAVING num_duplicate_rows > 1;

The query returned 615 duplicate records in the raw table — confirming that the raw export requires cleaning before it can be used for reliable analysis.

Validating the Clean Table

The data team had already produced a deduplicated version of the table called all_sessions. I ran a similar query against it — this time grouping only by the key business fields — to confirm it contained zero duplicate rows. The query returned no results, verifying the clean table's integrity.

This exercise reinforced a critical data engineering principle: never assume a dataset is clean. Even when a unique key is expected, validating it empirically before analysis is essential practice.

Task 3 — Analytical SQL Queries

With a validated, clean dataset, I moved on to extracting business insights through a series of progressively complex SQL queries.

Query 1: Visitor Volume & Unique Visitors

The first query established baseline traffic metrics — total product page views and the number of distinct visitors. Using COUNT(DISTINCT fullVisitorId) is critical here: it ensures each person is counted once regardless of how many sessions or page views they generated.

Query 2: Visitor Breakdown by Traffic Channel

I then segmented unique visitors by their channelGrouping — the acquisition source (Organic Search, Direct, Referral, Social, etc.). This kind of analysis is foundational for marketing teams deciding where to invest budget. The query returned 8 distinct channels.

Query 3: Product Catalogue & Deduplication with GROUP BY

To list all unique products, I used GROUP BY v2ProductName rather than SELECT DISTINCT. While both produce the same result, GROUP BY is the preferred pattern in BigQuery for large datasets as it integrates cleanly with aggregation in subsequent steps. The store had 633 distinct product names.

Query 4: Top 5 Most-Viewed Products

Filtering to type = 'PAGE' — to exclude non-page interaction types such as events and transactions — I counted views per product and ranked them descending.

BigQuery Standard SQL — Top Products by Views
SELECT
  COUNT(*) AS product_views,
  v2ProductName AS ProductName
FROM `data-to-insights.ecommerce.all_sessions`
WHERE type = 'PAGE'
GROUP BY v2ProductName
ORDER BY product_views DESC
LIMIT 5;

Advanced: Deduplicating Views with a CTE

The previous query counted every page view — including repeat visits by the same person. To measure unique viewer reach (each person counted once per product), I restructured the logic using a Common Table Expression (CTE). The CTE first generates one row per (visitor, product) pair, then the outer query aggregates those pairs.

BigQuery Standard SQL — CTE for Unique Views per Visitor
WITH unique_product_views_by_person AS (
  SELECT
    fullVisitorId,
    v2ProductName AS ProductName
  FROM `data-to-insights.ecommerce.all_sessions`
  WHERE type = 'PAGE'
  GROUP BY fullVisitorId, v2ProductName
)
SELECT
  COUNT(*) AS unique_view_count,
  ProductName
FROM unique_product_views_by_person
GROUP BY ProductName
ORDER BY unique_view_count DESC
LIMIT 5;

Query 5: Orders, Quantities & Average Per Order

Expanding further, I combined COUNT(productQuantity) — which counts only non-null rows, i.e. actual orders — with SUM(productQuantity) to get total units. Dividing the two produced the average units per order. This surfaced an interesting insight:

The 22 oz YouTube Bottle Infuser had the highest average order quantity at 9.38 units per order — suggesting bulk or gift purchasing behaviour, despite not being the most-viewed product.

Task 4 — Challenge Queries & Business Insights

The final task presented three open-ended analytical challenges, requiring me to combine multiple SQL concepts to answer real business questions.

Challenge 1: Conversion Rate Analysis

I calculated conversion rates for products where more than 1,000 units had been added to a cart or ordered — excluding frisbees (likely test/demo products). The conversion rate was defined as the ratio of order interactions to total product views.

The LOWER(v2ProductName) NOT LIKE '%frisbee%' pattern ensures case-insensitive exclusion, a common requirement when filtering text data that may not be consistently cased.

Challenge 2: Checkout Funnel Analysis with CASE

I mapped numeric action codes to descriptive labels using a CASE statement, then counted the distinct visitors at each stage of the purchase funnel. This is a classic funnel analysis pattern used in product analytics and growth teams.

BigQuery Standard SQL — Funnel with CASE Statement
SELECT
  COUNT(DISTINCT fullVisitorId) AS number_of_unique_visitors,
  eCommerceAction_type,
  CASE eCommerceAction_type
    WHEN '3' THEN 'Add product(s) to cart'
    WHEN '5' THEN 'Check out'
    WHEN '6' THEN 'Completed purchase'
    -- ... additional mappings
  END AS eCommerceAction_type_label
FROM `data-to-insights.ecommerce.all_sessions`
GROUP BY eCommerceAction_type
ORDER BY eCommerceAction_type;

Of the 56,010 visitors who added an item to their cart, 19,988 completed a purchase — a cart-to-purchase conversion rate of 35.68%. This means roughly 64% of high-intent shoppers abandoned their cart before checkout.

Challenge 3: High-Quality Abandoned Cart Identification

The most complex query of the lab: finding sessions where a high-quality visitor (session quality score above 60) added a product to their cart but generated zero revenue — the textbook definition of cart abandonment.

Two technical challenges made this interesting. First, there is no pre-built unique session ID — I had to construct one using CONCAT(fullVisitorId, CAST(visitId AS STRING)), because a single visitor can have multiple visits and visitId alone is not globally unique. Second, I used MAX(eCommerceAction_type) to find the furthest step each session reached in the funnel — if the maximum action was '3' (add to cart) and revenue was zero or null, the session was definitively an abandoned cart.

BigQuery Standard SQL — Abandoned Cart Detection
SELECT
  CONCAT(fullVisitorId,
    CAST(visitId AS STRING)) AS unique_session_id,
  sessionQualityDim,
  SUM(productRevenue) AS transaction_revenue,
  MAX(eCommerceAction_type) AS checkout_progress
FROM `data-to-insights.ecommerce.all_sessions`
WHERE sessionQualityDim > 60
GROUP BY unique_session_id, sessionQualityDim
HAVING
  checkout_progress = '3'
  AND (transaction_revenue = 0
    OR transaction_revenue IS NULL);

Key Takeaways

  • 1 Data validation before analysis is non-negotiable. The raw table contained 615 duplicate rows. Running a COUNT + GROUP BY + HAVING check before any analysis is a foundational habit that protects the integrity of every insight derived from the data.
  • 2 CTEs make complex logic readable and maintainable. Breaking the unique-views-per-visitor query into a two-stage CTE made the logic clear: first isolate unique pairs, then aggregate. This is how production-grade SQL should be written.
  • 3 COUNT vs COUNT DISTINCT matters enormously. Using plain COUNT(fullVisitorId) vs COUNT(DISTINCT fullVisitorId) produces fundamentally different numbers. Choosing the wrong one silently corrupts every downstream metric.
  • 4 Constructing surrogate keys is a real analytical skill. When no unique identifier exists (as with session data), combining fields with CONCAT and CAST to create a reliable composite key is essential — and prevents incorrect groupings.
  • 5 MAX() as a funnel progress indicator is an elegant pattern. Using MAX(eCommerceAction_type) per session to determine the furthest step reached — rather than checking for the presence of a specific step — is a clever and scalable approach to funnel analysis.
  • 6 BigQuery's scale changes the way you think about queries. With 21+ million rows processed in seconds at no infrastructure cost, BigQuery demonstrates why cloud-native data warehouses have replaced traditional databases for analytical workloads.
